# 22 QR — техническое задание (вторая опора проекта)

Документ **обязателен** к исполнению при реализации. Наряду с [`PROJECT_CONCEPT.md`](./PROJECT_CONCEPT.md) и [`DESIGN_LUMEN.md`](./DESIGN_LUMEN.md) задаёт **что** и **как** строить. Идея и смысл — в `PROJECT_CONCEPT`; **визуал Lumen** — в `DESIGN_LUMEN`; **маршруты, данные, состояние, критерии готовности** — здесь. При расхождении: **согласовать документы**, затем менять код.

**Аудитория кода:** один реальный пользователь, мобильный браузер, квартира, утро. Регистрации, бэкенд-аккаунты и «безопасность от взлома» **не** являются целью; защита — **от преждевременного показа контента** в нормальном сценарии сканирования.

---

## 1. Термины

- **Ввод** — экран большого QR, маршрут `/intro`. Не шаг 1…11.
- **Шаг *N*** — малый QR с индексом *N* ∈ 1…11, маршрут `/step/NN` (два десятичных знака: `01`…`11`).
- **Прогресс** — флажок «ввод пройден» + число **«завершено по цепочке до»** *K* (0…11). *K* = 0 после ввода, пока шаг 1 не отмечен завершённым.
- **Доступ к контенту шага** — HTML/текст/медиа шага *N* ренерятся **только** если проверка доступа прошла; иначе показывается **заглушка** без подгрузки полей контента шага (чтобы JSON/текст физически не попал в DOM «для глаз» из сети — лениво грузить JSON шага после `if`).

---

## 2. Стек и репозиторий

- **Vite** + **React 18** + **TypeScript**.
- **react-router** (v6) — маршрутизация.
- **Стили:** глобальные **CSS-переменные** по [`DESIGN_LUMEN.md`](./DESIGN_LUMEN.md) + **CSS Modules** для компонентов (или один слой стилей без параллельных систем). Визуал **не** отклонять от Lumen без правки `DESIGN_LUMEN.md`.
- **Контент:** JSON-файлы в репозитории; длинные тексты в JSON как строки (поддержка **Markdown** в поле `body` с рендером на клиенте лёгким парсером, например `react-markdown` — фиксировано при инициализации).
- **Сборка:** production — статические ассеты, **без** обязательного Node на хостинге. Подходит Netlify, Vercel, Cloudflare Pages, GitHub Pages (с `base` при необходимости).

---

## 3. URL, поведение корня, печать QR

| Назначение        | URL            | Печать на      |
|------------------|----------------|----------------|
| Ввод (большой)   | `/intro`       | Большой QR     |
| Шаг 1…11         | `/step/01` … `/step/11` | Соответствующий малый QR |

- **`/`** — **мгновенный** редирект на `/intro` (без лишнего интерактива; по желанию одна реплика «открой с телефона» — не обязательна).
- Для деплоя **в интернет**: схема **https**, фиксированный **домен**; ссылки в QR **не** с query-параметрами и **не** с хэш-фрагментами для логики (всё путь).
- **Основной деплой:** HTTPS в интернете — см. [`DEPLOYMENT.md`](./DEPLOYMENT.md) (GitHub Pages). Префикс пути приложения задаётся `base` в Vite (**`/22/`** для репозитория `22`; в QR закладывать полный публичный URL).
- Опционально локальный превью без облака: см. [`SERVER_LOCAL_MAC.md`](./SERVER_LOCAL_MAC.md).
- Стабильность: **после печати** URL **не** меняется. Правки — только **контент** в JSON/медиа, путь тот же.

## 4. Состояние в клиенте

- **Хранилище:** `localStorage`, один ключ, например `22qr.quest.v2` (версия в ключе; смена схемы или числа шагов — новый ключ + миграция не обязательна для pet-project, достаточно сброса).
- **Схема значения (JSON):**
  - `introDone: boolean` — пользователь **завершил** ввод (см. §5).
  - `completedThrough: number` — максимальный *N*, по которому нажата **кнопка завершения** шага (см. §5). Диапазон 0…11, старт **0** до первого прохождения шага 1.

**Сброс прогресса** (для отладки): в dev-режиме **скрытая** комбинация или кнопка внизу; в production **не** показывать. Для «сброса в день Икс» — пересобрать/залить чистый `localStorage` с устройства вручную или задокументировать cookie clear (достаточно устно владельцу).

## 5. Логика доступа (ядро)

### 5.1. Ввод

- Пока `introDone === false`: маршрут `/intro` показывает **полный** контент ввода из `content/intro.json` (или эквивалентной структуры).
- Любой `/step/NN` при `introDone === false` показывает экран **«Сначала ввод»** (тип заглушки **A**): без текста и медиа из шага *N*, без номера «правильного следующего шага» в явном виде.

**Завершение ввода:** одна первичная кнопка внизу экрана (например «Поняла, поехали» / по тону из концепта). По нажатию: `introDone = true`, `completedThrough` **не менять** (остаётся 0 до прохождения шага 1).

### 5.2. Шаги 1…11

Пусть `nextRequired = completedThrough + 1` (после ввода первый раз **1**).

При переходе на `/step/NN` с `introDone === true`:

| Условие | Экран |
|--------|--------|
| `N > completedThrough + 1` | Заглушка **«Рано»** (тип **B**): контент шага **не** рендерить, данные шага **не** подгружать в состояние до разрешения. |
| `N === completedThrough + 1` | **Активный** шаг: полный контент шага *N* из `content/steps/NN.json`. |
| `N <= completedThrough` | **Повтор** уже пройденного: тот же полный контент шага *N* (можно пометить бейджем «уже было» — опционально, по минимализму можно без). |

**Завершение шага:** внизу каждого **активного** шага (когда `N === completedThrough + 1`) — кнопка «Дальше» (или синоним). По нажатию: `completedThrough = max(completedThrough, N)`. **Навигация на следующий URL программно не обязательна** — пользователь идёт к следующему QR в квартире.

На **повторном** просмотре (`N <= completedThrough`) кнопку завершения снова пушить в прогресс **не** нужно (достаточно одного раза).

### 5.3. Инвариант анти-спойлер

До положительной проверки доступа для `/step/NN` приложение **не** должно вставлять в DOM контент `body`, подсказки подарка, медиа и заголовки шага *N*. Разрешена общая оболочка (фон бренда, прогресс-бар, общий заголовок приложения).

## 6. Типы контента шагов (данные)

В `content/steps/NN.json` поле **`kind`**:

- **`moment`** — тип A: эмоциональная карточка; обязательны минимум `title` и/или `body`; медиа опциональны.
- **`gift`** — тип B: плюс блок **`gift`** с полями:
  - `hintPoetic` — строка, слой 1;
  - `hintConcrete` — строка, слой 2;
  - `placePhoto` — опционально, путь `/media/...` к фото места.

**Жёсткие номера подарков:** шаги **3, 6, 9** — `kind: "gift"`; остальные по умолчанию `moment`, **кроме** шага **11** — отдельная финальная схема (§7).

---

## 7. Шаг 11 (финал)

В `content/steps/11.json`:

- Поле **`finale: true`** и структура **`credits`** — массив строк или слайдов (минимум: строки подряд с таймингом или по клику «дальше» между блоками).
- После титров — **`codaMd`:** короткий текст («кстати…»), **не** упоминающий айфон по бренду; поле **`physicalHint`** — ясная фраза к **физическому** действию в квартире (куда подойти / что открыть).
- При необходимости **одно** поле медиа для последнего кадра.

Визуальный шаблон финала отделять от обычного шага (типография киношная, но в рамках одного UI-кита).

---

## 8. Заглушки и копирайт ошибок

Три типа экранов без контента шага:

- **A** — до ввода (малый QR раньше большого).
- **B** — «рано» к конкретному `/step/NN` (перескок вперёд).

Тексты хранить **отдельным** файлом `content/system-messages.json`: пул вариантов для A и для B (не менее **5** каждого), выбор по **детерминированной** функции от `NN` или ротации, чтобы не было ощущения одного и того же отказа. Обязательные смысловые блоки — как в `PROJECT_CONCEPT` §10: доброта, «не настала очередь», намёк на прогресс в UI, лайфхак с маркером — **без** выдачи правильной перестановки.

## 9. Прогресс в UI

- Глобальный элемент: **11 сегментов** (или эквивалент), заполнение **1…completedThrough**; подпись **«Шаг X из 11»** где *X = completedThrough + 1* при активном следующем шаге, либо **«Продолжай с шага X»** — формулировка одна на весь проект.
- Отображать на `/intro` (лёгко) и на всех экранах шагов и заглушек.

## 10. Ввод: обязательные блоки контента

В `content/intro` (JSON + md-поля) отразить все пункты из `PROJECT_CONCEPT` §5: поздравление, наличие 11 точек, смягчение «не гонка / паузы», объяснение порядка без математики, опционально маркеры.

## 11. Медиа и пути

- Статика: каталог **`public/`** (или согласованный в Vite `public`).
- Рекомендуемая раскладка: `public/media/step-NN/` для фото и видео шага *N*; в JSON только **абсолютные пути веба** вида `/media/step-07/....`.
- Видео: `<video controls playsInline>`; **один** основной формат **MP4 (H.264)**; poster опционально.
- Изображения: **WebP** предпочтительно; **JPEG/PNG** допустимы.
- Любая замена файла по тому же пути **без** смены URL — допустима при доработке после «ядра».

## 12. UI/UX и внешний вид

Полная спецификация — [`DESIGN_LUMEN.md`](./DESIGN_LUMEN.md). Эталон глазами — [`design-lab/index.html`](../design-lab/index.html).

Кратко (всё подлежит Lumen, без альтернатив):

- **Мобильный первично:** вертикальная ориентация; safe-area для iOS.
- **Шрифты:** Cormorant Garamond (заголовки), DM Sans (UI); опционально Syne для мелких лейблов — как в `DESIGN_LUMEN`.
- **Фон:** многослойный градиент + лёгкий noise, не плоский однотон.
- **Карточки:** матовое стекло (`backdrop-filter`, токены `--lum-glass*`).
- **Подарочные шаги 3, 6, 9:** единый бейдж/мотив подарка, два слоя подсказки — как в эталоне лаборатории.
- **Кнопки:** primary + при необходимости ghost; полная ширина на узком экране там, где логика экрана требует одну главную CTA внизу.

Изменять палитру, шрифты или отказываться от стекла **запрещено** без явной просьбы владельца и правки `DESIGN_LUMEN.md`.

## 14. Критерии приёмки

1. Скан большого QR (открытие `/intro`) даёт полный ввод; кнопка выставляет `introDone`.
2. До `introDone` любой `/step/*` — заглушка **A**, контент шагов unavailable.
3. После ввода: `/step/01` открывает шаг 1; `/step/03` при `completedThrough < 2` — заглушка **B**, без утечки данных шага 3.
4. После «Дальше» на шаге 1: `completedThrough >= 1`, шаг 2 доступен как активный, шаг 1 — как повтор.
5. Шаги 3, 6, 9 рендерят два слоя подсказки подарка при `kind: "gift"`.
6. Шаг 11 — титры + `codaMd` + `physicalHint`; айфон **не** назван в текстах до этой логики (проверка глазами контента).
7. Прогресс **1…11** и подпись **X из 11** согласованы с `completedThrough`.
8. Lighthouse / ручная проверка: нет горизонтального скролла по умолчанию на iPhone; видео воспроизводится по тапу.

## 15. Порядок работ (фазы)

1. **Каркас:** Vite+React+TS, роуты `/`, `/intro`, `/step/:id`, state в `localStorage`, три шаблона экранов (ввод, шаг, заглушка A/B); стили с токенами **Lumen** из [`DESIGN_LUMEN.md`](./DESIGN_LUMEN.md), сверка с [`design-lab/index.html`](../design-lab/index.html).
2. **Логика:** реализовать §5 полностью, плейсхолдер-контент для 11 шагов.
3. **Прогресс-UI** и системные сообщения из пула.
4. **Шаблоны:** `moment`, `gift` (два слоя), `finale`.
5. **Наполнение:** тексты, фото, видео по местам; замена медиа без смены маршрутов.
6. **Деплой:** см. [`DEPLOYMENT.md`](./DEPLOYMENT.md); дополнительно опционально [`SERVER_LOCAL_MAC.md`](./SERVER_LOCAL_MAC.md); smoke-тест на реальном телефоне по HTTPS; печать QR по финальным URL.
